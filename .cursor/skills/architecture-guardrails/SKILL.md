# Architecture Guardrails

## Goal

Preserve a clean separation of responsibilities:

- `frontend` handles rendering and UX state.
- `backend` handles business logic, persistence, and external integrations.
- Shared contracts are explicit and type-safe.

## Layering Rules

1. Frontend does not directly embed provider credentials or integration secrets.
2. Backend is the single integration gateway for Deepgram, Groq, Gmail, Slack, and Calendar APIs.
3. Keep mock data isolated from API clients so migrations to real APIs are low-risk.
4. Avoid circular dependencies between feature modules.
5. New endpoints must expose stable request/response shapes.

## Data-Flow Rules

1. Recording and transcription flows should emit incremental updates.
2. AI outputs (summary/actions/decisions) should be generated by backend services, then rendered by frontend.
3. Ask-the-meeting queries should be routed through retrieval + generation pipelines, not static UI-only logic.

## Validation Checklist

- [ ] Feature touches only intended layer(s).
- [ ] API boundary is documented and testable.
- [ ] New service logic is prepared for persistence and retries.
- [ ] Security-sensitive logic is not placed in renderer-only code.
